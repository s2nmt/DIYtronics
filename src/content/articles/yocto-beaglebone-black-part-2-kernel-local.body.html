This is Part 2 of the Yocto on BeagleBone Black series. After Part 1 (image build and SD flash), this guide explains how to build the BeagleBone kernel from a local source tree using Yocto — how the upstream recipe and custom layer work together, and what happens at each stage.

1. Overview

By default, Yocto clones the kernel from GitHub into a temporary build directory on every fresh build. This project instead:

  • Keeps a local clone of the BeagleBoard.org kernel at linux-bb/
  • Uses a custom Yocto layer (meta-bbb) to tell BitBake to compile from that tree
  • Lets you edit kernel source directly and rebuild quickly

2. Project layout

yocto-bbb/
├── linux-bb/                                    ← local kernel source (edit here)
└── poky/
    ├── meta-ti/.../linux-bb.org_git.bb          ← upstream recipe
    ├── meta-ti/.../linux-bb.org/defconfig       ← defconfig pointer
    └── meta-bbb/.../linux-bb.org_%.bbappend    ← enables local build

Why meta-bbb? meta-bbb is a custom Yocto layer for this project. It holds kernel customizations without modifying upstream layers like meta-ti. If you update meta-ti, your changes in meta-bbb are preserved.

3. The base recipe: linux-bb.org_git.bb

Location: poky/meta-ti/meta-ti-bsp/recipes-kernel/linux/linux-bb.org_git.bb

This is the upstream recipe from TI's meta-ti layer.

SECTION = "kernel"
SUMMARY = "BeagleBoard.org Linux kernel"
LICENSE = "GPL-2.0-only"
COMPATIBLE_MACHINE = "beagle.*"

inherit kernel
require recipes-kernel/linux/setup-defconfig.inc
require recipes-kernel/linux/ti-kernel.inc

DEPENDS += "gmp-native libmpc-native"
KERNEL_EXTRA_ARGS += "LOADADDR=${UBOOT_ENTRYPOINT} ${EXTRA_DTC_ARGS}"

S = "${WORKDIR}/git"

SRCREV:armv7a = "4ca9ea30768d58c8d4d56d03dd1eaf8c8feb7ef9"
PV:armv7a = "6.1.80+git${SRCPV}"
BRANCH:armv7a = "v6.1.80-ti-r34"

SRC_URI = " \
    git://github.com/beagleboard/linux.git;protocol=https;branch=${BRANCH} \
    file://defconfig \
"

Key variables

VariableMeaning
inherit kernelStandard Yocto kernel build class
S = "${WORKDIR}/git"Source directory after git fetch
SRCREV:armv7aFixed commit for 32-bit ARM (BeagleBone Black)
SRC_URIFetches kernel from GitHub + a small defconfig file

The defconfig pointer file

Location: poky/meta-ti/meta-ti-bsp/recipes-kernel/linux/linux-bb.org/defconfig

use-kernel-config=bb.org_defconfig

This is not the full kernel config. It tells setup-defconfig.inc to use the in-tree defconfig:

linux-bb/arch/arm/configs/bb.org_defconfig

BitBake finds the pointer file because file://defconfig in SRC_URI resolves to the linux-bb.org/ directory next to the recipe — a standard Yocto convention.

4. The custom append: linux-bb.org_%.bbappend

Location: poky/meta-bbb/recipes-kernel/linux/linux-bb.org_%.bbappend

Why the filename matters

BitBake matches append files to recipes by name:

linux-bb.org_git.bb       ← base recipe (PN = linux-bb.org)
linux-bb.org_%.bbappend   ← append (% = wildcard for _git, _6.1, etc.)

If the name does not match, BitBake silently ignores the file.

Append vs overwrite

A .bbappend does not replace the base recipe. BitBake merges them. The base recipe structure stays intact; the append adds or overrides specific variables.

Full append file

FILESEXTRAPATHS:prepend := "${THISDIR}/${PN}:"

inherit externalsrc
EXTERNALSRC = "/home/user/workspace/yocto-bbb/linux-bb"
EXTERNALSRC_BUILD = "${WORKDIR}/build"

For a minimal local-kernel-only setup, that is all you need. The project may also include optional lines such as config fragments — those are separate from the core local-build mechanism.

Line-by-line

  • FILESEXTRAPATHS:prepend — tells BitBake where to find companion files (e.g. config fragments) in meta-bbb/recipes-kernel/linux/linux-bb.org/.
  • inherit externalsrc — enables building from an external source directory instead of cloning into WORKDIR.
  • EXTERNALSRC — absolute path to your local kernel tree (linux-bb/).
  • EXTERNALSRC_BUILD — separate directory for build artifacts (.o files, .config). Keeps object files out of your git tree.

To disable local build, comment out inherit externalsrc.

5. Kernel configuration flow

meta-ti/.../linux-bb.org/defconfig
    → use-kernel-config=bb.org_defconfig
        ↓
linux-bb/arch/arm/configs/bb.org_defconfig
    → base kernel .config
        ↓
(optional config fragments from meta-bbb)
        ↓
final .config in WORKDIR/build/

setup-defconfig.inc handles this in do_configure():

  1. Read the recipe's defconfig pointer
  2. Run make bb.org_defconfig against the kernel source
  3. Merge any KERNEL_CONFIG_FRAGMENTS if defined
  4. Run oldconfig to resolve dependencies

6. Local source with externalsrc

Normal flow (no externalsrc)

do_fetch     → git clone to WORKDIR/git
do_configure → configure in WORKDIR/git
do_compile   → build in WORKDIR/git

With externalsrc

do_fetch     → skip kernel clone (still fetches recipe companion files)
do_configure → source = linux-bb/,  build = WORKDIR/build
do_compile   → build in WORKDIR/build

Benefits

  • Edit kernel source directly in linux-bb/
  • Use normal git workflow in that directory
  • Rebuild quickly with bitbake linux-bb.org -c compile -f
  • No digging through tmp/work/ for source

Requirements

  • EXTERNALSRC must be an absolute path
  • Local tree should match the SRCREV the recipe expects

7. Setup and build

Step 1: Clone the kernel (one-time)

cd /path/to/yocto-bbb

git clone --branch v6.1.80-ti-r34 https://github.com/beagleboard/linux.git linux-bb
git -C linux-bb checkout 4ca9ea30768d58c8d4d56d03dd1eaf8c8feb7ef9

Step 2: Configure the append file

Ensure linux-bb.org_%.bbappend has inherit externalsrc and the correct EXTERNALSRC path. Add meta-bbb to bblayers.conf.

Step 3: Initialize the build environment

cd poky
source oe-init-build-env build-bbb

The build directory name (build-bbb, etc.) is arbitrary — each directory is an independent build environment.

Step 4: Build the kernel

bitbake linux-bb.org

Rebuild after source changes

bitbake linux-bb.org -c compile -f
bitbake linux-bb.org

Build output

poky/build-bbb/deploy-ti/images/beaglebone/
├── zImage
├── zImage-beaglebone.bin
├── am335x-boneblack.dtb
└── am335x-boneblack-beaglebone.dtb

8. How BitBake merges recipes

When you run bitbake linux-bb.org:

1. Parse linux-bb.org_git.bb
2. Merge linux-bb.org_%.bbappend (meta-bbb, higher priority)
3. Effective recipe:
   ├── Source:  EXTERNALSRC → linux-bb/  (overrides normal fetch for compile)
   ├── Config:  bb.org_defconfig
   └── Build:   inherit kernel (unchanged from base recipe)
4. Run tasks: fetch → configure → compile → install → deploy
SettingSource
Kernel version / SRCREVlinux-bb.org_git.bb (meta-ti)
inherit kernellinux-bb.org_git.bb
Defconfig pointerlinux-bb.org/defconfig (meta-ti)
Local source treelinux-bb.org_%.bbappend (meta-bbb)

9. Development workflow

  1. Edit files in linux-bb/
  2. bitbake linux-bb.org -c compile -f && bitbake linux-bb.org
  3. Check output in deploy-ti/images/beaglebone/
  4. Commit changes in linux-bb/ with git

Switching back to upstream fetch — comment out in .bbappend:

# inherit externalsrc

BitBake will clone from GitHub again on the next build.

10. Troubleshooting

Append not taking effect

  • Check filename: linux-bb.org_%.bbappend
  • Confirm meta-bbb is in bblayers.conf
  • Run: bitbake-layers show-appends linux-bb.org

externalsrc path error

  • EXTERNALSRC must be absolute
  • Directory must exist before building

Clean rebuild

bitbake linux-bb.org -c cleansstate
bitbake linux-bb.org

Summary

FileRole
linux-bb.org_git.bbUpstream recipe — version, fetch, build class
linux-bb.org/defconfigPoints to bb.org_defconfig in kernel tree
linux-bb.org_%.bbappendEnables local build via externalsrc
linux-bb/Your local kernel source — edit and rebuild here

Standard Yocto practice: keep upstream layers untouched, put customization in meta-bbb, use .bbappend to extend the recipe, and use externalsrc for fast kernel iteration.
