This is Part 3 of the Yocto on BeagleBone Black series. It explains how to bring up a Sitronix ST7789V TFT display using Yocto — device tree, kernel modules, rootfs packages, and optional boot splash. Assumes you already have a local kernel build with meta-bbb and externalsrc.

1. Overview

The ST7789 is a common small TFT controller, often wired over SPI. On Linux, the standard approach on embedded boards is:

  1. Describe the display in the device tree
  2. Use the in-kernel fbtft framework and fb_st7789v driver
  3. Expose the screen as a framebuffer device: /dev/fb0
  4. Let userspace (your app or a boot splash script) write pixels to that device

This project does not use a custom display driver in userspace. The kernel handles SPI communication; your application only needs to write to the framebuffer.

2. How the display stack works

  ST7789 panel (hardware)
         │
         │  SPI1 + GPIO (DC, RST)
         ▼
  Device Tree node (sitronix,st7789v)
         │
         ▼
  Kernel modules: fbtft + fb_st7789v
         │
         ▼
  /dev/fb0  (framebuffer)
         │
         ▼
  Userspace: app, fbset, boot splash script
LayerResponsibility
Device treePin mux, SPI bus, GPIO for DC/RST, driver binding

UserspaceDraw to /dev/fb0

Kernel logs stay on UART (ttyS0). The display is reserved for your application — not for printk output.

3. Hardware connections

In this project, the display is attached to SPI1 on the BeagleBone Black expansion header:

SignalBBB PinNotes
SCKP9_31SPI1 clock
MOSIP9_29SPI1 data out
CSP9_28SPI1 chip select 0
DCP9_27Data/Command (GPIO)
RSTP9_25Reset (GPIO)
MISOP9_30Not used by this display

Important: The fbtft driver binds directly to SPI1. Do not add a spidev@0 node on the same chip select — the two drivers would conflict.

4. Device tree configuration

All display hardware is described in the local kernel tree.

File: linux-bb/arch/arm/boot/dts/am335x-bone-common.dtsi

Pin mux

Two pin groups are defined under &am33xx_pinmux:

  • spi1_pins — SPI1 bus lines (SCK, MOSI, MISO, CS)
  • st7789_gpio_pins — DC on P9_27, RST on P9_25

SPI1 and display node

&spi1 {
    #address-cells = <1>;
    #size-cells = <0>;
    status = "okay";
    pinctrl-names = "default";
    pinctrl-0 = <&spi1_pins &st7789_gpio_pins>;
    ti,pindir-d0-out-d1-in;

    fbtft@0 {
        compatible = "sitronix,st7789v";
        reg = <0>;
        spi-max-frequency = <20000000>;
        fps = <30>;
        buswidth = <8>;
        rotate = <90>;
        dc-gpios = <&gpio3 19 GPIO_ACTIVE_HIGH>;   /* P9_27 */
        reset-gpios = <&gpio3 21 GPIO_ACTIVE_LOW>; /* P9_25 */
    };
};
PropertyValueMeaning
compatiblesitronix,st7789vBinds the fb_st7789v driver
spi-max-frequency20000000SPI clock: 20 MHz
buswidth88-bit SPI
fps30Target refresh rate
rotate90Rotate output 90° — adjust if orientation is wrong
dc-gpiosgpio3 pin 19Data/Command line
reset-gpiosgpio3 pin 21Hardware reset

After editing the device tree, rebuild the kernel:

bitbake linux-bb.org -c compile -f
bitbake linux-bb.org

Deploy the updated zImage and am335x-boneblack.dtb to the board.

5. Kernel configuration

The BeagleBoard.org defconfig already enables the required drivers as modules.

File: linux-bb/arch/arm/configs/bb.org_defconfig

CONFIG_FB_TFT=m
CONFIG_FB_TFT_ST7789V=m

No extra kernel config is needed if you build with bb.org_defconfig.

Disable framebuffer console

If the kernel printed its boot log to the TFT, it would overwrite anything your application draws. This project disables that via a config fragment.

File: meta-bbb/recipes-kernel/linux/linux-bb.org/no-fbcon.cfg

# Console on UART only — ST7789 used for splash/app, not kernel log
# CONFIG_FRAMEBUFFER_CONSOLE is not set
# CONFIG_LOGO is not set

Merge it in linux-bb.org_%.bbappend:

SRC_URI:append = " file://no-fbcon.cfg"
KERNEL_CONFIG_FRAGMENTS:append:beaglebone = " ${WORKDIR}/no-fbcon.cfg"

Result: kernel console on UART only; /dev/fb0 is free for your use.

6. Yocto layer changes

All display-related Yocto customization lives in meta-bbb.

Module autoload

File: meta-bbb/recipes-kernel/linux/linux-bb.org_%.bbappend

KERNEL_MODULE_AUTOLOAD:append:beaglebone = " fbtft fb_st7789v"

At boot, the kernel loads fbtft (SPI TFT framework) and fb_st7789v (ST7789 driver), which probes the device tree node and creates /dev/fb0.

Install modules in the image

File: meta-bbb/recipes-core/image/core-image-minimal.bbappend

IMAGE_INSTALL:append = " \
    kernel-module-fbtft \
    kernel-module-fb-st7789v \
    fbset \
"
PackagePurpose
kernel-module-fbtftSPI TFT framework module
kernel-module-fb-st7789vST7789 panel driver
fbsetInspect and configure framebuffer parameters

Without these packages, the modules exist only in the kernel build output — they are not on the rootfs and modprobe will fail on the board.

7. Building and deploying

Build kernel (device tree + modules)

cd poky
source oe-init-build-env build-bbb
bitbake linux-bb.org

Build full image (modules + rootfs)

bitbake core-image-minimal

Output files

build-bbb/deploy-ti/images/beaglebone/
├── zImage
├── am335x-boneblack.dtb
└── core-image-minimal-beaglebone-*.rootfs.wic

Flash the .wic image to SD card or eMMC, or deploy zImage + .dtb manually if you only changed the kernel.

8. Verifying on the board

After boot, SSH or serial into the board and run:

# Check modules loaded
lsmod | grep -E 'fbtft|st7789'

# Check framebuffer exists
ls -l /dev/fb0

# Read display info
fbset -i

# Check kernel probe messages
dmesg | grep -iE 'fbtft|st7789|fb0'

If /dev/fb0 is missing, try loading manually:

modprobe fbtft
modprobe fb_st7789v

A successful probe typically shows a line like:

graphics fb0: fb_st7789v device registered, ...x... buffer

Quick pixel test

If you have a raw RGB565 image matching the panel resolution (e.g. 320×240 = 153600 bytes):

dd if=logo.raw of=/dev/fb0 bs=153600 count=1

The screen should update immediately.

9. Optional: boot splash

This project includes a bbb-boot-splash recipe that shows a logo at boot.

How it works:

  1. Systemd starts bbb-boot-splash.service early in boot
  2. show-boot-splash.sh loads fbtft and fb_st7789v if needed
  3. Waits up to 30 seconds for /dev/fb0 to appear
  4. Writes logo.raw (320×240 RGB565, 153600 bytes) to the framebuffer

Key files:

meta-bbb/recipes-core/bbb-boot-splash/
├── bbb-boot-splash_1.0.bb
└── files/
    ├── show-boot-splash.sh
    ├── bbb-boot-splash.service
    └── logo.raw

The recipe is added to core-image-minimal.bbappend via bbb-boot-splash in IMAGE_INSTALL. No extra steps are needed if you build the full image.

11. Summary

Enabling the ST7789 on BeagleBone Black with Yocto requires four things:

StepWhatWhere
1Describe hardware in device treelinux-bb/.../am335x-bone-common.dtsi
2Enable fbtft + fb_st7789v in kernel configbb.org_defconfig (already set)
3Disable kernel fbconno-fbcon.cfg in meta-bbb
4Autoload modules and install into imagelinux-bb.org_%.bbappend, core-image-minimal.bbappend

Once /dev/fb0 exists, any userspace program can draw to the display — no proprietary SDK required. The kernel handles SPI; you write pixels.
