This is Part 5 of the Yocto on BeagleBone Black series. It explains how to bring up the InvenSense MPU6050 6-axis IMU (accelerometer + gyroscope). Unlike button-p8-keys, there is no custom driver — the board uses the standard in-kernel IIO driver (inv-mpu6050). Your job is to wire it correctly, describe it in the device tree, enable the kernel modules, and read sensor data through the IIO sysfs interface.

1. Overview

The MPU6050 communicates over I2C. Linux exposes it through the Industrial I/O (IIO) subsystem — not a custom sysfs driver like the P8 buttons.

ItemDetail
SensorMPU6050 (accel + gyro)

Compatible stringinvensense,mpu6050

No separate recipe is needed for the driver itself — only device tree changes, kernel modules in the image, and optional i2c-tools for debugging.

2. How the sensor stack works

  MPU6050 chip (hardware)
         │
         │  I2C2 (SDA, SCL)
         ▼
  Device Tree node (invensense,mpu6050 @ 0x68)
         │
         ▼
  Kernel modules: inv-mpu6050-i2c + inv-mpu6050
         │
         ▼
  IIO subsystem
         │
         ▼
  /sys/bus/iio/devices/iio:deviceN/
      ├── in_accel_x_raw, in_accel_y_raw, in_accel_z_raw
      ├── in_anglvel_x_raw, in_anglvel_y_raw, in_anglvel_z_raw
      └── in_temp_raw (if exposed)

The I2C driver (inv-mpu6050-i2c) handles bus communication. The core driver (inv-mpu6050) registers accelerometer and gyroscope channels with the IIO framework.

3. Hardware connections

In this project, the MPU6050 is on I2C2:

SignalBBB PinNotes
SCLP9_19I2C2 clock
SDAP9_20I2C2 data
VCC3.3Ve.g. P9_03 or P9_04
GNDGNDe.g. P9_01 or P9_02

I2C address

AD0 pinAddress
GND0x68 (used in this project)
VCC0x69

The device tree uses reg = <0x68>. If your module has AD0 tied high, change it to <0x69>.

Pull-ups

I2C requires pull-up resistors on SDA and SCL (typically 4.7 kΩ to 3.3V). Many breakout boards include them. The device tree enables internal pull-ups on the AM335x pins, but external pull-ups on the breakout are usually still required for reliable operation.

4. Device tree configuration

File: linux-bb/arch/arm/boot/dts/am335x-bone-common.dtsi

I2C2 pin mux

i2c2_pins: i2c2-pins {
    pinctrl-single,pins = <
        AM33XX_PADCONF(AM335X_PIN_UART1_CTSN, PIN_INPUT_PULLUP, MUX_MODE3)  /* P9_20: i2c2_sda */
        AM33XX_PADCONF(AM335X_PIN_UART1_RTSN, PIN_INPUT_PULLUP, MUX_MODE3)  /* P9_19: i2c2_scl */
    >;
};

I2C2 bus and IMU node

&i2c2 {
    pinctrl-names = "default";
    pinctrl-0 = <&i2c2_pins>;
    status = "okay";
    clock-frequency = <100000>;

    /* MPU6050: P9_19=SCL, P9_20=SDA, addr 0x68 */
    imu@68 {
        compatible = "invensense,mpu6050";
        reg = <0x68>;
    };
};
PropertyValueMeaning
status = "okay"—Enable I2C2 controller
clock-frequency100000I2C bus speed: 100 kHz (standard mode)
compatibleinvensense,mpu6050Binds the inv-mpu6050 IIO driver
reg<0x68>I2C slave address

After editing the device tree, rebuild the kernel:

bitbake linux-bb.org -c compile -f
bitbake linux-bb.org

Deploy the updated am335x-boneblack.dtb to the board.

5. Kernel configuration

The BeagleBoard.org defconfig already enables the required drivers as modules.

File: linux-bb/arch/arm/configs/bb.org_defconfig

CONFIG_IIO=y
CONFIG_INV_MPU6050_I2C=m

CONFIG_INV_MPU6050_I2C automatically selects the core inv-mpu6050 module. No extra config fragment is needed in meta-bbb for the MPU6050.

Module names

KconfigModule file
CONFIG_INV_MPU6050_IIOinv-mpu6050.ko (core, pulled in as dependency)
CONFIG_INV_MPU6050_I2Cinv-mpu6050-i2c.ko

Both must be present on the rootfs for I2C probing to work.

6. Yocto layer changes

All MPU6050-related Yocto settings are in meta-bbb.

Module autoload

File: meta-bbb/recipes-kernel/linux/linux-bb.org_%.bbappend

KERNEL_MODULE_AUTOLOAD:append:beaglebone = " inv-mpu6050 inv-mpu6050-i2c"

At boot, the kernel loads both modules so the driver can probe the I2C device automatically.

Install modules and tools in the image

File: meta-bbb/recipes-core/image/core-image-minimal.bbappend

IMAGE_INSTALL:append = " \
    i2c-tools \
    kernel-module-inv-mpu6050 \
    kernel-module-inv-mpu6050-i2c \
"

8. Verifying on the board

Step 1: Check I2C bus

# List I2C buses
ls /dev/i2c-*

# Scan I2C bus 2 for devices
i2cdetect -y 2

You should see 68 (or 69) in the grid:

Step 2: Check kernel modules

lsmod | grep inv_mpu

Expected:

inv_mpu6050_i2c    ...
inv_mpu6050        ...

If missing:

modprobe inv-mpu6050-i2c
modprobe inv-mpu6050

Step 3: Check kernel probe message

dmesg | grep -i mpu

A successful probe typically shows the driver binding to the I2C device.

Step 4: Check IIO device

ls /sys/bus/iio/devices/
cat /sys/bus/iio/devices/iio:device*/name

You should see device names like mpu6050 or similar for accel/gyro channels.

9. Reading sensor data

IIO exposes each channel as a sysfs file. Find your device first:

IIO_DEV=$(grep -l mpu /sys/bus/iio/devices/iio:device*/name 2>/dev/null | head -1)
IIO_DEV=$(dirname "$IIO_DEV")
echo "Device: $IIO_DEV"

List available channels

ls $IIO_DEV/

Typical files:

in_accel_x_raw    in_accel_y_raw    in_accel_z_raw
in_anglvel_x_raw  in_anglvel_y_raw  in_anglvel_z_raw
in_temp_raw
in_accel_scale    in_anglvel_scale

Read accelerometer (raw values)

cat $IIO_DEV/in_accel_x_raw
cat $IIO_DEV/in_accel_y_raw
cat $IIO_DEV/in_accel_z_raw

Read gyroscope (raw values)

cat $IIO_DEV/in_anglvel_x_raw
cat $IIO_DEV/in_anglvel_y_raw
cat $IIO_DEV/in_anglvel_z_raw

Convert to physical units

Raw values must be multiplied by the scale factor:

# Example: read scale and compute
SCALE=$(cat $IIO_DEV/in_accel_scale)
RAW=$(cat $IIO_DEV/in_accel_x_raw)
echo "$RAW * $SCALE" | bc -l

Simple monitoring loop

while true; do
    echo -n "accel: "
    echo -n "$(cat $IIO_DEV/in_accel_x_raw) "
    echo -n "$(cat $IIO_DEV/in_accel_y_raw) "
    echo -n "$(cat $IIO_DEV/in_accel_z_raw)  "
    echo -n "gyro: "
    echo -n "$(cat $IIO_DEV/in_anglvel_x_raw) "
    echo -n "$(cat $IIO_DEV/in_anglvel_y_raw) "
    echo "$(cat $IIO_DEV/in_anglvel_z_raw)"
    sleep 0.5
done

For production applications, consider libiio or direct read() on /dev/iio:deviceN with proper buffer setup for higher sample rates.

10. Troubleshooting

SymptomLikely causeFix
i2cdetect shows -- at 0x68Wiring, power, wrong busCheck SDA/SCL, 3.3V, GND; confirm bus number 2
Device at 0x69 not 0x68AD0 pin tied highChange reg = <0x69> in device tree
modprobe failsModules not in imageAdd kernel-module-inv-mpu6050* to IMAGE_INSTALL
I2C seen but no IIO deviceModules not loaded, wrong compatibleLoad modules; check compatible string
Probe error in dmesgWrong address, bus speed, powerTry clock-frequency = <400000>; verify hardware
Multiple IIO devicesOther sensors on same busIdentify by name file in each iio:deviceN

Useful debug commands

# Device tree node present?
ls /proc/device-tree/soc/i2c@481a0000/imu@68/ 2>/dev/null
cat /proc/device-tree/soc/i2c@481a0000/imu@68/compatible

# I2C bus status
ls /sys/bus/iio/devices/

# Read chip WHO_AM_I register directly (MPU6050 = 0x68)
i2cget -y 2 0x68 0x75

Register 0x75 (WHO_AM_I) should return 0x68 on a genuine MPU6050.

1Enable I2C2 and describe the sensorlinux-bb/.../am335x-bone-common.dtsi
2Enable INV_MPU6050_I2C in kernel configbb.org_defconfig (already set)
3Autoload kernel moduleslinux-bb.org_%.bbappend

Unlike button-p8-keys, there is no custom driver or separate recipe. The MPU6050 uses the standard in-kernel inv-mpu6050 driver. Device tree tells the kernel where the chip is; IIO sysfs gives you accelerometer and gyroscope data in userspace.
